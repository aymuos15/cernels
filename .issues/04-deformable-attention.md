# 04 Â· Multi-scale deformable attention (DETR family)

**What it is.** Multi-scale deformable attention (MSDeformAttn) is the core op of the DETR detection/segmentation family â€” Deformable DETR, Mask2Former, OneFormer, RT-DETR, D-FINE. Instead of dense attention over all spatial positions, each query attends to a small set of *learned sampling points* across multiple feature-map scales: for every (query, head, level, point) it bilinearly samples the value map at a data-dependent location and sums the samples weighted by attention weights. transformers marks it kernel-swappable via `@use_kernel_forward_from_hub("MultiScaleDeformableAttention")` â€” 10 models use it.

**Why compile loses.** The reference (`multi_scale_deformable_attention` in transformers) loops over levels calling `F.grid_sample` per level, then stacks and does a weighted sum â€” it materializes a per-level sampled tensor `(B*heads, hidden, num_queries, points)` for every level and reduces over them. That's gather-heavy with data-dependent sampling coordinates; `torch.compile` can't fuse the bilinear sampling + cross-level weighted reduction into a single pass, and grid_sample is already an opaque ATen op. The canonical fix is the original Deformable-DETR CUDA kernel: one fused pass that bilinearly samples, multiplies by the attention weight, and accumulates â€” no per-level intermediates.

**Baseline â€” library reference (per [setting up baselines](../../docs/guide/setting_up_baselines.md)).** baseline = transformers' `multi_scale_deformable_attention` (the pure-torch grid_sample reference; e.g. `transformers.models.deformable_detr.modeling_deformable_detr`). One-line call, all tensors built in `inputs()`. **Do not hand-write it.** `lib` = the Hub kernel `kernels-community/deformable-detr` (layer `MultiScaleDeformableAttention`, Version 1, resolves on the Spark: `torch212-cu130-aarch64` compatible âœ…). **Config sketch.** `Config` (non-Hub baseline + Hub `lib` + `custom`), dtype fp32 (detection models run fp32 here; bilinear sampling needs the precision). `op = "multi_scale_deformable_attention"`. `inputs(device, dtype)` builds: `value` `(B, L_total, n_heads, hidden)`, `value_spatial_shapes` (per-level H,W), `sampling_locations` `(B, num_queries, n_heads, n_levels, n_points, 2)` in [0,1], `attention_weights` `(B, num_queries, n_heads, n_levels, n_points)` (softmax-normalized over levels*points). All raw tensors â€” nothing precomputed that the op should do. `verify` at atol ~1e-3 (fp32 bilinear; match grid_sample's `align_corners=False`, `padding_mode="zeros"`, and the `2*loc-1` grid convention exactly). `use_compile = True` (the grid_sample reference compiles but stays gather-bound â€” that's the point).

**Inputs to think about.** Deformable-DETR defaults: `B=2`, `n_heads=8`, `hidden=32` (so model dim 256), `n_levels=4`, `n_points=4`, `num_queries=300`. Multi-scale value length = sum of H*W over 4 levels, e.g. `(92,92),(46,46),(23,23),(12,12)` â†’ `L_total â‰ˆ 11633`. Big enough that the per-level grid_sample intermediates dominate.

**Difficulty.** Mediumâ€“high. The CUDA kernel itself is well-trodden (the upstream MSDeformAttn kernel is a known reference), but two things are fiddly: (1) reproducing `grid_sample`'s exact bilinear/border semantics so `verify` passes against the torch baseline, and (2) wiring the layer-style Hub `lib`. Keep the sampling+reduction in the timed path; `inputs()` only builds raw value/locations/weights.
