# 07 · RoI Align (SAM3 box-prompt pooling)

**What it is.** Region-of-Interest Align: given a feature map and a set of boxes, it crops each box to a fixed `output_size × output_size` grid by **bilinearly sampling** `sampling_ratio²` points per output cell and averaging them. It's the standard detection/segmentation pooling op (Mask R-CNN origin) and is used in SAM3's geometry/box-prompt path (`torchvision.ops.roi_align` in `transformers/models/sam3/modeling_sam3.py`, pooling vision features per prompt box to a `roi_size=7` grid).

**Why a custom kernel can win.** RoI Align is a gather-heavy bilinear-sampling op — the same family as the deformable-attention kernel we already made beat its upstream lib by ~1.5×. `torch.compile` treats `torchvision.ops.roi_align` as an opaque C++ op (no fusion), so the real bar is **torchvision's own CUDA roi_align**. A fused single-pass kernel (one thread per output element, bilinear-sample the `sampling_ratio²` sub-points with fp32 accumulate, no intermediate tensors) can match or beat it — exactly as our deformable kernel did against its tuned upstream.

**Baseline — built-in library op (per [setting up baselines](../../docs/guide/setting_up_baselines.md)).** baseline = `torchvision.ops.roi_align` (the op *is* the reference, like `nms`). `lib` = none (torchvision is the baseline). `custom` = our CUDA roi_align — reuse the bilinear-sampling machinery from `src/kops/registry/deform_attn.cu` (same `px = ... ; bilinear interp ; align_corners` care). **Match torchvision's exact semantics:** `aligned`/`align_corners` convention (torchvision default `aligned=False`: pixel coord = `x * spatial_scale`, no −0.5 shift unless aligned), `sampling_ratio` (default −1 → adaptive `ceil(roi_size / output_size)`), and the average pooling over sample points.

**Config sketch.** `Config` (non-Hub, torchvision is the baseline like `nms`). dtype fp16 (torchvision roi_align supports fp16/fp32; SAM3 casts bf16→fp16 for it). `op = "torchvision.ops.roi_align"`. `inputs(device, dtype)` builds raw: feature map `value (B, C, H, W)`, boxes (either a `(K, 5)` tensor with batch-index col, or a list of `(n_i, 4)` per image — match torchvision's accepted form), plus scalars `output_size`, `spatial_scale`, `sampling_ratio`, `aligned`. `baseline` = one `torchvision.ops.roi_align(...)` call. `custom` takes the same inputs. `verify` at atol ~1e-3 (fp16 bilinear; match the sampling-grid + averaging exactly). `use_compile = True` (compile just calls the opaque op — it won't beat the custom).

**Inputs to think about.** SAM3-ish: feature map `(B=2, C=256, H=72, W=72)`, `num_boxes≈128` total (random valid xyxy boxes in-frame), `output_size=7`, `spatial_scale=1.0`, `sampling_ratio=2`, `aligned=True`. Big enough that the bilinear gathers dominate.

**Difficulty.** Medium. The kernel is well-trodden (Mask R-CNN roi_align), and we have the bilinear-sampling code from deformable attention. The fiddly parts: matching torchvision's `aligned`/`sampling_ratio` semantics so `verify` passes, and the box-format plumbing (`(K,5)` vs list). Keep sampling+pooling in the timed path; `inputs()` only builds the feature map + boxes.

**Refs.** Mask R-CNN (He et al. 2017) for RoI Align; torchvision `roi_align` source for exact semantics. SAM3 usage: `transformers/models/sam3/modeling_sam3.py` (box pooling). Reuse `src/kops/registry/deform_attn.cu` bilinear sampling.
